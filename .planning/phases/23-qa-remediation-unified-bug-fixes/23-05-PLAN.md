    **Fix 1 — WCAG contrast (QAR-13):**
    Read `frontend/app/globals.css` and audit all CSS variable values for contrast ratio against their backgrounds:

    Focus on these variables in both `:root` and `[data-theme="dark"]`:
    - `--text-secondary`: Must have >= 4.5:1 ratio against `--surface` background
    - `--text-tertiary`: Must have >= 4.5:1 ratio against `--surface` background
    - `--text-muted` (if exists): Same requirement
    - Any other low-contrast text variables

    Common fix: If `--text-secondary` is something like `#999999` on `#FAFAF7` (contrast ratio ~2.8:1), change to `#666666` or `#737373` for ~4.6:1 ratio.

    For dark mode: If `--text-secondary` is too dim on the dark background, increase brightness (e.g., from `#888` to `#A0A0A0`).

    Use these reference ratios (WCAG AA for normal text = 4.5:1):
    - Light mode bg #FAFAF7: text needs to be #767676 or darker for 4.5:1
    - Dark mode bg #1A1816: text needs to be #949494 or lighter for 4.5:1

    Also check `--border` and any interactive element colors. Adjust as needed.

    **Fix 2 — Landscape bottom nav (QAR-15):**
    Read `frontend/components/NavLayout.tsx` or whatever component renders the mobile bottom nav (check MobileTabBar). Find any height-based media query or JS logic that hides the nav bar.

    ```css
    @media (orientation: landscape) and (max-height: 500px) {
      /* Ensure bottom nav remains visible */
    }
    ```

    If the hide logic is in a React component (e.g., checking `window.innerHeight < X`), adjust the threshold to account for landscape mode at 844px width devices.

    **Fix 3 — Silent message drop during streaming (QAR-18):**
    Read `frontend/components/ChatContainer.tsx`. Find the `handleSendMessage` function (line ~680). The current guard is:
    ```tsx
    if (!messageToSend.trim() || isStreaming) return
    ```
    This silently drops the user's message. Replace with a queued message approach:
    1. Add state: `const [queuedMessage, setQueuedMessage] = useState<string | null>(null)`
    2. When `isStreaming` is true and user sends a message, store it: `setQueuedMessage(messageToSend)` and show a visual notice (e.g., a small toast or inline text below the input: "Will send after response completes")
    3. Add a `useEffect` that watches `isStreaming`: when it transitions from `true` to `false` and `queuedMessage` is not null, auto-send the queued message and clear the queue
    4. Clear `queuedMessage` when user clears input or navigates away
    5. Also apply the same pattern to `handleSuggestionClick` (line ~643) — show notice instead of silently returning

    Alternatively, if queuing is too complex, the minimum fix is to show a visual notice: change the `return` to display a brief toast/inline message like "Please wait for the current response to finish" using the existing toast/notification pattern in the codebase. Do NOT silently drop the message.

    ```tsx
      redirect('/')
    }
    ```
    This converts a "silent unexplained redirect" into a "documented intentional redirect."
